Warum 87% der KI-Projekte scheitern: Die Ontologie-Lücke

Das teuerste Problem in der KI hat nichts mit Modellen zu tun. Es ist die Frage, die niemand stellt: Was SIND eigentlich unsere Daten?
Florian Ziesche 23. Februar 2026 8 min

Letzte Woche habe ich ein politisches Intelligence-System gebaut. 13 Versionen in einem Tag. Die ersten 11 Versionen waren Feature-getrieben: "Mach das klickbar", "Füg eine Heatmap hinzu", "Zeig Verbindungen." Version 12 war der Wendepunkt — und der Grund, warum ich diesen Artikel schreibe.

Was in Version 12 passiert ist: Ich habe aufgehört, Features zu bauen. Und angefangen, meine Daten zu definieren.

Was eine Ontologie ist (und warum Sie eine brauchen)

Eine Ontologie beantwortet drei Fragen:

Die drei Fragen
Was KANN existieren?
→
Welche Eigenschaften hat es?
→
Wie hängt es zusammen?

Palantir — das Unternehmen, das für CIA, US Army und mittlerweile BMW und Airbus arbeitet — hat sein gesamtes Geschäftsmodell auf einer Ontologie gebaut. Nicht auf Dashboards. Nicht auf KI-Modellen. Auf der Definition, was ein "Objekt" in ihrem System ist.

Wenn es gerendert wird, ist es ein Objekt. Wenn es ein Objekt ist, hat es eine Definition. Wenn es keine Definition hat, gehört es nicht ins System.

Das €4,4 Milliarden-Problem

Die EU verliert jedes Jahr €4,4 Milliarden an Fördermitteln. Nicht weil die Programme schlecht sind. Sondern weil die Daten in den Verwaltungen nicht miteinander reden.

✗ Ohne Ontologie
  • Fördermittel in Excel
  • Fristen im Kalender
  • Zuständigkeiten im Kopf
  • Bürgerbeschwerden per Mail
  • Jedes System eigene Sprache
✓ Mit Ontologie
  • Fördermittel = Objekt mit Frist, Betrag, Status
  • Frist = verknüpft mit Zuständigkeit + Projekt
  • Zuständigkeit = Person → Abteilung → Gemeinde
  • Beschwerde = verknüpft mit Thema → Projekt → Frist
  • Ein System, eine Sprache

Warum Feature-First ein Anti-Pattern ist

Mein Intelligence-System hatte nach 11 Versionen:

28
klickbare Elemente
6
Tab-Ansichten
0
konsistente Datenstruktur

Jeder onclick-Handler war manuell geschrieben. Jede Farbe war ad-hoc gewählt. Jeder Datenbank-Eintrag hatte ein anderes Schema. Es funktionierte — für 5 Personen. Es wäre bei 50 zusammengebrochen.

Version 12 führte eine einzige Funktion ein: obj(). Sie wrappt jeden Datenpunkt automatisch in ein klickbares Element. Wenn der Datenpunkt keine Definition hat, bekommt er einen roten Rand — sichtbar für den Entwickler, unsichtbar für den Nutzer.

// Vorher: Ad-hoc, pro Element <span onclick="showDetail('protschka-vs')">VS-Relevanz</span> // Nachher: Ontologie-gesteuert, automatisch obj('VS-Relevanz', 'protschka-vs') // → Rendet klickbar wenn Definition existiert // → Rendet rot-gepunktet wenn Definition FEHLT // → Audit-Log zählt Coverage automatisch

Eine Zeile statt drei. Und die wichtigste Eigenschaft: Lücken werden sichtbar, nicht versteckt.

Die 5 Gesetze einer guten Ontologie

Aus 80+ Projekten (BMW, Siemens, Bosch, und jetzt Kommunen) habe ich 5 Regeln destilliert:

1. Jedes Datenelement existiert genau einmal
Duplikation = Inkonsistenz = Vertrauensverlust. Wenn eine Kontroverse in 3 Tabellen steht, gibt es 3 Versionen der Wahrheit.
2. Jedes Objekt hat 7 Felder. Nicht mehr, nicht weniger.
Titel, Typ, Beschreibung, Evidenz, Konfidenz, offene Lücken, verwandte Objekte. Ob es eine Person ist, eine Kontroverse oder eine Fördermittel-Frist — das Schema ist identisch.
3. Lücken sind Features
Jedes Objekt muss sagen, was es NICHT weiß. "Keine Daten, die Kausalität belegen" ist ehrlicher und nützlicher als eine halluzinierte Erklärung.
4. Schema vor Code
Erst definieren was existieren KANN, dann bauen wie es aussieht. Nicht umgekehrt. 11 Versionen Features, dann die Ontologie — das ist zu spät.
5. Auto-Generate > Manual Maintain
Wenn ein Detail-Eintrag automatisch aus bestehenden Daten generiert werden kann, darf er nicht manuell gepflegt werden. Manuelle Pflege skaliert nie.

Was das für Ihr nächstes KI-Projekt bedeutet

Bevor Sie das nächste Dashboard bauen, die nächste KI-Integration planen, oder den nächsten Chatbot installieren — beantworten Sie eine Frage:

Was sind die 10 wichtigsten Objekte in Ihrem System? Haben sie alle das gleiche Schema?

Wenn die Antwort "nein" ist, haben Sie kein KI-Problem. Sie haben ein Ontologie-Problem. Und kein Modell der Welt wird das für Sie lösen.

Die 87% der KI-Projekte, die scheitern, scheitern nicht an GPT-4, Claude oder Gemini. Sie scheitern daran, dass niemand die erste Frage gestellt hat: Was SIND eigentlich unsere Daten?
